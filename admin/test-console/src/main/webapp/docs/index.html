Test Console Documentation

One toolkit, two call sources: the BLADE test apps replace SIPp with scenario-driven testing that lives in the cluster itself.

The pieces

  • test-uac — the originator. Its load engine synthesizes INVITEs at a target CPS or concurrency, per node. As a B2BUA it can also transform real softphone calls (e.g. dress them up as SIPREC recorder legs).
  • test-uas — the terminator. It answers calls with scripted response sequences (ringing, rejections, transfers, auto-teardown) or forwards them while transforming the message (e.g. stripping SIPREC multipart down to bare SDP).
  • This console — discovers every tester node's TesterControl MBean over federated JMX, fans out start/stop/reset, and aggregates each node's live metrics.

Scenarios

A scenario describes what happens to a call: a role (originate / answer / b2bua), an optional template (a raw SIP message fragment merged into the INVITE), transformation rules (the CRUD engine's rule sets — headers, MIME parts, SDP/XML/JSON edits), a response script, and assertions. Scenarios live in test-uac.json and test-uas.json — edit them schema-validated in the Configurator.

On the terminator, a call selects its scenario with a ;scenario=name Request-URI parameter, a translation-plan match, or the classic shorthands (;status=486, ;delay=5s, ;refer=sip:…), which keep working with no configuration at all.

Running a load test

  1. Pick the app (usually test-uac) and, optionally, a scenario name and pacing overrides. Empty fields use the configuration's originate defaults.
  2. Start on all nodes sends the run to every matching node — targets are per node, so 3 nodes × 100 CPS = 300 CPS aggregate.
  3. Watch the node cards (engine state, active calls, counters) and the scenario table (status distribution, latency percentiles, expectation mismatches, assertion tallies). Counters are summed across nodes; latency columns show the worst node.
  4. Stop drains active calls naturally. Reset metrics clears the per-scenario counters for a fresh measurement window.

Per-call variables & assertions

Each originated call exposes ${lastStatus}, ${statusSequence}, ${setupMs}, ${index}, ${scenario}, plus anything a rule's read operation captured. Assertions are boolean expressions over those variables — ${lastStatus} == 200 && ${setupMs} < 500 — tallied per scenario as pass / fail / warn.

Equivalent REST (per node)

Everything the console does is also available per node on the service tier:

POST /test-uac/api/v1/loadtest/start   {"scenario":"load-basic","mode":"cps","targetCps":100}
POST /test-uac/api/v1/loadtest/stop
GET  /test-uac/api/v1/loadtest/status
GET  /test-uac/api/v1/loadtest/report
POST /test-uac/api/v1/loadtest/reset

The console itself uses federated JMX (never REST) to reach the nodes, per BLADE admin-tier convention.
