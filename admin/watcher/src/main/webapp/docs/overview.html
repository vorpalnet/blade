Overview

The pump pattern

Every BLADE service that exposes configuration registers a SettingsMXBean at vorpal.blade:Name=<app>,Type=Configuration. To change a service's config at runtime you need to:

  1. Write the new JSON to disk under config/custom/vorpal/.
  2. Find the matching MBean.
  3. Stream the JSON into it.
  4. Tell it to reload.

Steps 2–4 are mechanical. That's what the "pump" is. Watcher is one implementation of the pump. The Configurator includes its own implementation of the same logic.

Which pump should I deploy?

Configurator is deployed
You don't need Watcher. The Configurator's ConfigurationMonitor covers the pump role.
Configurator is not deployed, but you still want file → MBean updates
Deploy Watcher. Useful for headless setups where operators edit JSON files via a different mechanism (SSH, Ansible, deploy pipeline) and want the runtime to pick them up automatically.
Configurator AND Watcher are both deployed
Both will pump. Not harmful — the second pump's update is a no-op since the file hasn't changed by the time it picks it up. Slightly wasteful but operationally fine.

What it doesn't do

  • No UI — there's no page to interact with.
  • No validation — it pumps whatever JSON is on disk, valid or not. Bad JSON shows up as a SettingsMXBean reload exception.
  • No editing — that's the Configurator's job.
Future

Watcher is "Legacy" because the consolidation of the pump into the Configurator made the standalone deployment optional. It's kept around because there is no harm in keeping a tiny zero-UI WAR available — but new deployments rarely need it.
