Edit Sessions

The WLS edit-and-activate cycle

Every change to the WLS domain configuration goes through an explicit cycle:

  1. Start an edit session. WLS gives you a writable view of the configuration. Only one edit session per domain at a time.
  2. Modify attributes. Your changes apply only inside the edit view; the live runtime sees nothing yet.
  3. Save. Validates the pending changes against WLS's internal rules. Failure here means the changes are syntactically wrong (e.g. negative heap size) — no harm done.
  4. Activate. Promotes the saved changes to the live runtime. WLS applies them, possibly with a delay for restart-required attributes.

If anything fails before Activate, you can Undo — discards the edit session entirely.

How the dashboard threads this

The dashboard wraps each Save action into a complete start → modify → save → activate cycle. Operator intent is "I changed this field; persist it" — they shouldn't have to think about edit sessions. Behind the scenes:

  1. Browser POSTs the changed value(s).
  2. Backend opens an edit session via the WLS Edit MBean server.
  3. Backend writes the attributes to the matching ConfigurationMBean(s).
  4. Backend saves and activates.
  5. Backend returns success; UI clears the dirty state.

The lock problem

WLS allows only one edit session at a time. If another operator is editing via the WLS Admin Console and hasn't activated or undone, every Save from the dashboard will fail with an "Already in edit session" error.

Stealing the lock

WLS supports forcibly stealing an edit lock — but only do this if you're certain the other session is stale (operator left their browser open overnight). Stealing discards their pending changes irrecoverably. Coordinate first.

What "restart-required" means at activation time

Activate succeeds for both runtime and restart- required attributes. The difference is:

  • Runtime attributes take effect immediately on the live MBean.
  • Restart-required attributes are written into the persisted config.xml but the running JVM keeps its current value until that server is restarted.

WLS reports restart-required after activation via the ServerLifeCycleRuntime MBean. The dashboard surfaces this as a yellow banner so you know a planned restart is needed.

Multi-server attributes

Many tuning attributes are per-server (each server has its own heap, its own work manager). For those, the dashboard shows the value separately per server and lets you Save per-server or "Save to all". Per- server granularity is essential when a deployment has heterogeneous hardware.
