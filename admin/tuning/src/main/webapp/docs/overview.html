Overview

What it is

A curated dashboard for the WLS / OCCAS tuning knobs that BLADE operators most often need to touch. Every knob on the dashboard corresponds to a real WLS MBean attribute. The dashboard is a thin presentation layer: read on page load, write on Save.

What it isn't

  • A replacement for the WLS Admin Console — that remains the source of truth for the full WLS configuration surface.
  • A configuration store — values are read from / written to the live WLS configuration. The dashboard doesn't keep its own copy.
  • A monitoring tool — for live metrics (CPS, session counts, GC stats) see your platform's monitoring stack.

The sections

JVM
Heap min/max, GC selection, GC log destination, key flags. Restart-required.
SIP Timer
T1, T2, T4, timer F, expiration intervals — the timer table that governs SIP retransmission and dialog lifecycle. Restart-required.
JDBC Pools
Per-data-source Initial / Min / Max connection capacity. Min and Max resize on activate; Initial Capacity is restart-required.
Work Manager
Min/max threads, capacity constraints, fair-share request classes — how WLS schedules work across CPU.
Cluster
Cluster membership view (read-only) and the unicast vs. multicast knobs for cluster heartbeat.
Server Tuning
Per-server execute queues, socket receive/send buffer sizes, accept backlog, native I/O toggles.

Why curated

WLS exposes thousands of MBean attributes. The vast majority of them are either WLS-internal, BLADE-irrelevant, or already at sensible defaults. The dashboard collects the handful of knobs BLADE operators actually reach for and presents them with the right framing (units, ranges, runtime-vs-restart warnings).

How writes happen

WLS configuration is edited through an explicit edit session: start-edit → modify attributes → save → activate. The dashboard threads this for every Save automatically. See Edit Sessions for the mechanics and the failure modes.

Recommended presets

The JVM, Server Tuning, OCCAS Threads, and JDBC Pools sections each have a Recommended button. It only pre-fills the standard high-CPS tuning values — heap -Xms = -Xmx, Max Message Size ×4, wlss.timer Max Threads ≥ 200 with Min = Max where both constraints exist, and JDBC Initial = Min = Max (≥ 300). Nothing is written until you review the filled-in values and Save, so the preset is a starting point, not an auto-apply.

Restart-required knobs

Some knobs (heap size, native I/O, listen ports) take effect only on server restart. The dashboard marks these explicitly with a yellow warning. The Save will succeed and the new value will be in config.xml, but the running JVM keeps the old value until it restarts.
