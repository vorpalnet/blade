Troubleshooting

Symptom-to-cause table

SymptomLikely causeWhat to do
Save fails with "Already in edit session" Another operator has an open WLS edit session Wait, or coordinate. As last resort, steal the lock from the WLS Admin Console — see Edit Sessions.
Save succeeds but the running server still shows the old value Restart-required attribute; the new value is in config.xml but the JVM keeps the old one Plan a rolling restart at a maintenance window. The dashboard shows a yellow banner for these.
Read returns a stale value (different from what was just saved by another tool) Brief cache window in WLS; usually self-corrects within 10s Wait and refresh. If persistent, restart the dashboard WAR — the cached MBean handle may be stale.
Dashboard fails to load any values JMX walk failed — usually the WAR isn't on the AdminServer Verify deployment target. tuning.war belongs on the AdminServer, not the cluster.
Save returns success but the value reverts on next read An automated tool (Ansible, deploy pipeline) is overwriting your change shortly after Check for orchestration that re-applies config — coordinate with whoever owns it.
Validation error on Save — value rejected by WLS The value violates a WLS internal constraint (negative number, value out of range, malformed expression) Error message names the offending attribute and constraint. Adjust and re-save.
Tuning under load

Avoid tuning under live production load when possible. Many runtime-tunable attributes (thread pool sizes, queue depths) take effect immediately and can destabilize an in-flight service. Tune during low-traffic windows or in staging first.
