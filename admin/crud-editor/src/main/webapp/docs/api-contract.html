API Contract

Base path

All endpoints are rooted at /<app>/resources/api/. The editor derives <app> from the page URL.

Endpoints

MethodPathPurpose
GET{base}/{resource}List all rows of the named resource.
GET{base}/{resource}/{key}Fetch one row by primary key.
PUT{base}/{resource}/{key}Create or update one row by primary key. Body is the row JSON.
DELETE{base}/{resource}/{key}Delete one row.
POST{base}/{resource}/_previewDry-run preview of a mutating operation. Body describes the proposed change; response describes the effect without applying it.

Response shape

List responses:

{
  "resource": "rules",
  "rows": [
    { "key": "...", "field1": "...", "field2": ... },
    ...
  ],
  "schema": { /* JSON schema describing row shape */ }
}

The optional schema field lets the editor render typed form fields; without it, the editor falls back to free-form text editing.

Dry-run preview

Mutating operations (PUT, DELETE) optionally route through POST {base}/{resource}/_preview first. The preview response describes the expected effect: 

{
  "action": "update",
  "key": "row-42",
  "before": { ... },
  "after":  { ... },
  "diff":   [ { "field": "...", "from": ..., "to": ... } ],
  "warnings": [ "..." ]
}

The editor renders the diff in a confirmation modal before sending the actual mutating request. This is how an operator can sanity-check a change before committing — particularly valuable for table-shaped data where one bad row can affect every call that matches.

Authentication

All endpoints inherit the BLADE admin tier's session auth. No separate token; the same login that opens the portal opens the CRUD API.

Server-side responsibility

This page documents what the editor expects. The actual REST handler lives in each BLADE service. If a service doesn't implement these endpoints, the editor will get 404s — that's the signal that the service hasn't opted into CRUD-API editability.
