Overview

Configuration vs. resources

BLADE services have two kinds of editable state:

Configuration
One JSON document per service — settings, defaults, thresholds. Edited by the Configurator.
Resources
Table-shaped data — multiple rows. Each row is independent. Examples: ACL entries, routing rules, customer records, blocked-number lists. Edited by the CRUD Editor.

The line between them is a design choice per service. If a service exposes a small number of well-defined knobs, those go in the Configurator. If it owns a growing list of similarly-shaped rows, those go in CRUD.

Which services expose CRUD

Any BLADE service can mount the CRUD REST API. The editor doesn't have a pre-baked list — it discovers the API base from the page URL and points there. Add a new service that implements the contract, browse to /blade/crud-editor/?app=<new-service>, and the editor will work without changes.

What it doesn't do

  • Bulk import/export (use the service's own API directly for that).
  • Cross-service joins (the editor sees one resource table at a time).
  • Schema authoring (the form is generated from the API response shape, not edited here — for schema authoring use Configurator).

How the URL works

The editor's base API URL is computed from location.pathname: the first path segment becomes the app name, and the API base becomes /<app>/resources/api/. This means the editor doesn't need configuration to know which app to edit — it picks up the app from where it's deployed (or from a query string, when iframed through the portal).

Same WAR, many apps

The editor WAR is generic. The browser-side script derives the target API base from the URL. So one CRUD Editor deployment can edit any service's resources just by varying the URL.
