Cluster Discovery

The JMX walk

Server discovery is a four-step walk through the federated DomainRuntime MBean server. The viewer's ClusterDiscovery class does it on every page load (cached briefly):

  1. JNDI lookup java:comp/env/jmx/domainRuntime → MBeanServer reference.
  2. Resolve com.bea:Name=DomainRuntimeService,Type=...DomainRuntimeServiceMBean → service ObjectName.
  3. Read its DomainConfiguration attribute → DomainMBean ObjectName (note: not "Domain" — the attribute is literally DomainConfiguration).
  4. Read Servers from the DomainMBean → array of ServerMBean ObjectNames.

For each ServerMBean, the viewer reads Name, ListenAddress, ListenPort, and the linked Cluster MBean's Name. That tuple populates the dropdown.

The DomainConfiguration trap

On WLS 14.1.1, the literal lookup com.bea:Name=DomainConfiguration,Type=Domain throws InstanceNotFoundException. The correct path is to start at DomainRuntimeService and read its DomainConfiguration attribute (which resolves to the actual DomainMBean ObjectName).

What's listed

Every server defined in the WLS domain — AdminServer plus every managed/engine-tier server — appears in the dropdown, whether it's currently running or not. The viewer doesn't filter by reachability; an unreachable server's log requests will return errors at read time.

Why a server might be missing

SymptomCause
Dropdown is empty The JMX walk failed entirely. Check AdminServer log for ClusterDiscovery exceptions.
One server missing It's defined in config.xml but the WLS Servers attribute didn't enumerate it — extremely rare; almost always means the WLS config is malformed.
Server appears but log requests fail with "no such MBean" The per-server LogReader MBean isn't registered on that server. The server is up but no BLADE WAR has started (or every WAR's LogReaderRegistrar failed). See Troubleshooting.

The per-node MBean side

For each server that appears in the dropdown to actually be readable, that server must have at least one BLADE-framework WAR deployed and active. The framework's LogReaderRegistrar (in libs/framework) is wired via web-fragment.xml into every WAR — so any service WAR's deployment is enough. The registration is idempotent across multiple WARs on the same JVM.

Inspecting MBeans directly

To verify the chain works, query the federated DomainRuntime MBeanServer for vorpal.blade:Type=LogReader,Name=* from the AdminServer side — you should see one MBean per active node. If a node is missing, that's where the problem lives.
