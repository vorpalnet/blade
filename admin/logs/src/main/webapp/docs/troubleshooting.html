Troubleshooting

Symptom-to-cause table

SymptomLikely causeWhere to look
Server dropdown is empty JMX walk failed — usually means the page isn't running on the AdminServer (no DomainRuntime there) Verify the deploy target. logs.war belongs on the AdminServer, not the engine-tier cluster.
Server dropdown has servers, but one is missing WLS config doesn't list it, or the federation hasn't caught up after a recent topology change WLS Admin Console → Servers. If it's listed there but not here, restart this WAR; the discovery cache may need refresh.
Selected a server, log dropdown is empty That server has no LogReader MBean — no BLADE-framework WAR is deployed/active on it Verify a BLADE WAR is actually running on that server. Even a single service WAR is enough — the framework wires the listener via web-fragment.xml automatically.
Log dropdown populated, read returns 500 / "MBean operation failed" The remote MBean threw — probably a file-permission issue on the target server Check the target server's stdout for stack traces from LogReaderClient / VorpalLogReader.
Read returns blank body but no error offset=-1 tail of a file shorter than the requested size, plus EofReached=true Normal. The file is shorter than the tail you asked for; you're seeing all of it.
Tail mode shows old content; new lines don't appear WLS log buffering — the file isn't flushed yet WLS flushes on its own schedule (configurable per server). For high-urgency real-time, lower the flush interval in the server's logging config, or read the diagnostic logs which flush more eagerly.
HTTP 401 / login redirect on every API call Session expired — every API call is auth-protected by the same realm as the page Re-login via the portal.

Sanity checks

Is my MBean registered on a given server?

From the AdminServer, query the federated DomainRuntime MBeanServer:

vorpal.blade:Type=LogReader,Name=<serverName>

One result = MBean is there. Zero = the listener didn't register on that server.

Is the AdminServer's domain runtime tree healthy?

Query com.bea:*Domain* from a JMX client on the AdminServer. You should see DomainRuntimeService and the federated DomainConfiguration. If those are missing, the federation is broken (rare; usually means WLS itself is in an unhealthy state).

Reading huge files

The slice mechanism handles arbitrary file sizes, but the response body is the slice bytes — asking for max=10000000 sends 10 MB through the AdminServer to your browser per request. Keep max reasonable (a few MB at most) for production sanity.
