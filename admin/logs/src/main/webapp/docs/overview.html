Overview

The problem this solves

In a clustered OCCAS deployment, every engine-tier server writes its own log files into its own per-server directory. When an operator needs to investigate an incident — "what did the proxy-router on cluster1_server3 log at 14:32?" — the traditional answer is to SSH to the host, find the domain home, navigate to the per-server logs directory, and tail the file. Multiply by N hosts and the operator's time investment grows linearly with cluster size.

This viewer collapses the workflow into a single browser page. The AdminServer holds the UI and the REST endpoints; each cluster node exposes its log directory via a per-JVM MBean; WebLogic's DomainRuntime MBeanServer federates the per-node MBeans so the AdminServer can read any of them.

The three pieces

Per-node MBean
Every WAR that bundles the framework JAR registers a LogReader MBean via the framework's LogReaderRegistrar @WebListener. The MBean is named vorpal.blade:Type=LogReader,Name=<serverName>, so each server has exactly one. The MBean exposes a small API: list logs and read a slice of a named log file.
WLS federation
WLS automatically federates every engine-tier server's platform MBeanServer into the AdminServer's DomainRuntime MBeanServer. The AdminServer can query the federated tree by ObjectName pattern and invoke MBean operations on remote servers transparently — no manual JMX URL, no credentials beyond the WLS user the page is authenticated as.
The viewer REST + UI
This WAR's LogQueryAPI exposes three JAX-RS endpoints: list servers, list a server's logs, read a slice. The HTML page calls them, populates the server / log dropdowns, and tails the chosen file. The implementation is in blade/admin/logs/.

Why this design

  • No SSH — operators don't need shell access to every cluster host.
  • No log shipper — no Filebeat, no Fluentd, no copy-to-central indexer. Logs stay where the server wrote them.
  • No per-host paths in code — the MBean knows its own server's log directory, so adding a new server requires no viewer config change.
  • Same auth as the rest of the admin tier — protected by the same WLS realm / FORM auth as every other admin WAR.

What's NOT in scope

  • Aggregation — there's no merged-across-servers view. Pick one server at a time.
  • Indexed search — the read is byte-offset slicing, not a search index. For full-text search across history, point Splunk / Loki / OpenSearch at the same files on disk.
  • Real-time streaming — the viewer polls; it doesn't hold a long-lived WebSocket. Adequate for incident triage; not adequate for a wall-of-logs dashboard.
Hot path

The viewer is intentionally read-only. There's no "clear log" or "rotate now" — those are server-side concerns owned by WLS log rotation policy.
