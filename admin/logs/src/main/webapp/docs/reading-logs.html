Reading Log Files

Listing logs

Once a server is selected, the viewer calls GET api/servers/{name}/logs. The response is a list of log file descriptors: 

[
  {
    "relativePath": "AdminServer.log",
    "sizeBytes": 1284733,
    "lastModifiedMs": 1742480123000,
    "kind": "WLS_SERVER"
  },
  {
    "relativePath": "AdminServer-diagnostic.log",
    "sizeBytes": 92340,
    "lastModifiedMs": 1742479000000,
    "kind": "WLS_DIAGNOSTIC"
  },
  ...
]
relativePath
Path relative to the server's log directory. The viewer uses this as the URL path component for reads — slashes are allowed (and the JAX-RS path is {path:.+} to accept them).
sizeBytes
Current file size. Used to compute "where is the end" for tail reads.
lastModifiedMs
Last-modified time. Useful for sorting most-recently-written first.
kind
Coarse-grained classification (server log, access log, diagnostic, GC log, etc.). Drives UI grouping or filtering if added later.

Reading a slice

The read endpoint is GET api/servers/{name}/logs/{path:.+}?offset=N&max=M. The response body is plain text (the bytes of the slice as UTF-8). Three response headers describe the slice:

X-Log-NewOffset
Byte offset to use for the next read to continue from where this one left off. Use this for follow-mode polling.
X-Log-EofReached
true if the slice reached the current end of file.
X-Log-TruncatedAtStart
true if the requested slice was clipped at the start (e.g. offset=-1 tail of a file smaller than max).

Tail mode

Pass offset=-1 to read the last max bytes of the file. This is the standard "tail -c" semantics. The response's X-Log-NewOffset header gives you the current EOF position — use it for subsequent follow-mode reads.

Why bytes, not lines

Lines are a UI concern; the wire protocol is bytes. This keeps the slice deterministic and avoids edge cases like "how do you tail N lines of a file with 300-character lines?" The viewer renders bytes as text in the browser, where a trailing partial line is visible but harmless.

Encoding

Reads assume UTF-8. Logs that contain raw bytes outside the UTF-8 range will show replacement characters in the viewer but the byte slice itself is faithful.

Follow-mode loop

A polling follow-mode is straightforward:

  1. Read with offset=-1, max=<tail-size> to seed the viewer.
  2. Note X-Log-NewOffset from the response.
  3. Wait briefly (1–3s).
  4. Read again with offset=<newOffset>, max=<big-enough>.
  5. If response body is non-empty, append; update newOffset.
  6. Loop.
