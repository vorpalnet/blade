See the code behind the call

A callflow is how a BLADE application expresses a whole SIP conversation. Instead of the traditional JSR-289 model — where the logic for one call is scattered across separate doInvite, doResponse, doAck, and doBye handlers that you have to mentally reassemble — a callflow is a single process(...) method you read top to bottom, the same order the messages happen.

The lambda-continuation model

The trick is that each response handler is a lambda passed to the request that provokes it. You call sendRequest(request, response -> { ... }), and the lambda runs when that response arrives; inside it you sendResponse(response, ack -> { ... }), and so on. The conversation nests exactly as it flows: 

sendRequest(bobInvite, bobResponse -> {
    SipServletResponse aliceResponse = aliceInvite.createResponse(bobResponse.getStatus());
    copyContentAndHeaders(bobResponse, aliceResponse);
    sendResponse(aliceResponse, aliceAck -> {
        sendRequest(copyContentAndHeaders(aliceAck, bobResponse.createAck()));
    });
});

That is a full back-to-back call bridge. The response lambda even fires more than once — once for 180 Ringing, again for 200 OK — without you writing a state machine. Behind the scenes the framework serializes the continuation into the SIP session, so a paused callflow survives node failover across the cluster.

The Source tab is the real code

When you pick a traced step, the Source tab shows the real source of the app that handled it — copied out of the framework and service tree when the app was built, not a hand-written snippet — with the exact line that sent the message marked. So it can't drift from the code that actually runs, and it works the same for framework flows, BLADE services, and customer-built apps.

Callflow vs. Flow

The Flow editor shows inter-app routing — which application FSMAR hands a call to. The Traces tool shows intra-app conversation — what each application does once it's handling the call. Flow is the map; a callflow is the trip.

Traces — the flight recorder

The Traces page is the heart of this tool. Arm a rule (match a header pattern, cap the capture count), reproduce the call, and read the recording: the SIP messages coming and going through every BLADE app in the chain — raw message text included — merged into one timeline, with each sent message pinned to the exact source line that emitted it. Arming is in-memory runtime state on the Trace MBeans — it never touches configuration and disappears on restart.

The detail panel on the right has three tabs — Sequence, Message and Source. Picking a call opens the Sequence tab; picking a step updates all three but leaves you on whichever tab you are viewing. The Sequence tab draws the same steps as a ladder diagram — one lifeline per party (each BLADE app boxed, the external softphones dashed) and one arrow per message hop, in time order. A message crossing an app is logged twice — the app that sends it and the next app that receives it — and the two are matched by Call-ID (the reliable per-hop key) into a single arrow between their two lanes; a routed call lays out caller → app₁ → … → appₙ → callee and the responses flow back. The external softphone names come from the From/To headers only at the true ends of the chain (they are end-to-end, so they can't name an internal hop). The timeline and the diagram are linked both ways: click a step on the left and its hop lights up; click a hop and its step is selected, with the raw message and source line following along.

Snapshots — sharing what you saw

Save Snapshot downloads the captured calls as a single self-contained .html file: the trace data, the source file behind every step, and this viewer's own rendering are all embedded. Anyone can open it in a browser — no OCCAS, no login, no network — and see the same timeline and marked source lines you saw. Attach it to a ticket or email; it stays readable long after the trace buffer is gone.

Load Snapshot opens a saved file back in the live viewer (Refresh returns to live data) — the saved file also opens directly on its own, so loading is only needed when you want it side-by-side with live captures.

Snapshots carry source code

A snapshot embeds the application source for every traced step. Share it with the same care as the code itself.

Read-only

This tool never changes configuration — it only displays source and traces, and trace arming is transient runtime state.
