Operator Handbook

BLADE Files edits domain files that have no JSON Schema — the ones the Configurator can't manage. It is deliberately narrow: it only touches files an administrator has explicitly registered, and it never lets the browser browse the filesystem.

The editable-file registry

The set of editable files lives in this app's own settings, config/custom/vorpal/files.json — a list of entries, each with a label, a path (relative to DOMAIN_HOME), and a type (XML, PROPERTIES, or TEXT). Edit that list in the Configurator (the files app) — it has a JSON Schema, so it gets a validated form there. This is deny-by-default: a file that isn't in the registry cannot be opened or saved, period.

Path confinement

Every registered path is resolved against DOMAIN_HOME and canonicalized. A path that resolves outside the domain (via .. or an absolute path) is rejected server-side, even if it's in the registry. The server never trusts a path from the browser that isn't an exact match for a registry entry.

Validation on save

  • XML — parsed for well-formedness (with external entities disabled); a malformed document is rejected and the file on disk is left untouched.
  • Properties — parsed with java.util.Properties; a malformed escape or line is rejected.
  • Text — saved as-is, no structural check.

These are well-formedness checks, not semantic ones. A well-formed sipserver.xml can still be wrong for the SIP container — the tool guarantees the file parses, not that its contents are correct.

Backups & restore

Before any overwrite, the current file is copied into a sibling .versions/ directory as <name>.<timestamp>.version. The most recent 20 versions per file are retained. The editor's version dropdown lists them; Restore brings one back as the live file (backing up the current content first). This is the same backup discipline the Configurator uses, shared via the framework's VersionedFileStore.
