Command-Line Validation

The Configurator ships a REST API that validates every app's config files against their JSON Schema, and optionally deploys them to the running cluster (propagate via JMX + reload). blade-validate.sh, in the module root, is a thin wrapper over that API for use in CI pipelines and scripted rollouts.

REST API

The endpoints live under the Configurator's context root, /blade/configurator/api/v1/:

Method & pathDoes
GET /api/v1/validateValidate every discovered app's config files against its .jschema.
GET /api/v1/validate/{app}Validate one app's configs.
POST /api/v1/deployValidate all; if all pass, propagate to the cluster via JMX and reload.
POST /api/v1/deploy/{app}Validate one app; if it passes, propagate and reload it.
POST /api/v1/publish/{app}Push one app's config to the cluster and reload, without the validate gate.

Validation walks all three target scopes (domain, cluster, server) and reports each file's result. deploy only writes a tier over JMX when the on-disk file differs from what the node already has (timestamp check), so a shared-filesystem cluster is a no-op rather than a needless rewrite.

Authentication — HTTP Basic, not the login form

The browser Editor authenticates with the FORM login page and the shared admin-tier session cookie. Command-line clients can't drive a login form, so the /api/v1/* paths are authenticated separately: pass credentials in the standard Authorization: Basic header.

How it works

/api/v1/* is carved out of the WAR's FORM security-constraint and authenticated by an in-WAR filter (BasicAuthFilter) that validates the header against the same WebLogic realm and requires one of the admin roles (Admin, Operator, Deployer, Monitor). The browser app is untouched — it still uses FORM login. A request with no valid credentials gets 401 (with a WWW-Authenticate challenge), not the login page.

Call it directly with curl:

curl -u 'weblogic:secret' \
     http://localhost:7001/blade/configurator/api/v1/validate

The wrapper: blade-validate.sh

blade-validate.sh [options]

  --deploy           Validate and deploy (propagate + reload via JMX)
  --app <name>       Target a specific app (default: all apps)
  --host <host:port> AdminServer host (default: localhost:7001)
  --user <user>      WebLogic username (default: weblogic)
  --password <pass>  WebLogic password (required)
  --context <path>   Configurator context root (default: blade/configurator)

Examples:

# Validate everything
blade-validate.sh --password secret

# Validate one app
blade-validate.sh --app proxy-registrar --password secret

# Validate and roll out all configs to the cluster
blade-validate.sh --deploy --password secret

# Against a remote AdminServer
blade-validate.sh --deploy --app transfer \
    --host admin.example.com:7001 --password secret

The script authenticates with the Basic header, pretty-prints the JSON response, and exits non-zero when any result reports "valid": false or "deployed": false — so it gates a CI step directly.

Passwords with shell metacharacters

Wrap the password in single quotes so the shell doesn't expand !, $, etc. Inside single quotes nothing is escaped — write the literal password, no backslashes: --password '!Secret$1', not "\!Secret\$1".

Reading the output

Each file checked appears in results[] with its app, level (domain / cluster/<name> / server/<name>), and either "valid": true or a "valid": false with an errors list of schema messages. An app with a schema but no config files reports level: "none" and is valid (it runs on its sample/default). The top-level valid is the AND of every file.
