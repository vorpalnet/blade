Schema Discovery

The contract

Each BLADE service that wants to appear in the Configurator does one thing in its startup code: instantiate a SettingsManager with a Settings POJO class and (optionally) a sample. The framework does the rest. 

// In any service's @WebListener or SIP servlet contextInitialized:
settingsManager = new SettingsManager<>(event, OptionsSettings.class, new OptionsSettingsSample());

That single line causes the framework to:

  1. Generate a JSON Schema from the Settings POJO using victools/jsonschema-generator + Jackson annotations.
  2. Write SCHEMA.json and SAMPLE.json for the app under config/custom/vorpal/samples/<app>/.
  3. Register a SettingsMXBean at vorpal.blade:Name=<app>,Type=Configuration on the JVM's platform MBean server.
  4. Federate that MBean into the OCCAS domain runtime MBean server (WLS handles this automatically).

How the editor finds it

When the Schema dropdown is opened, the Configurator queries the OCCAS DomainRuntime MBean server for every MBean matching the pattern: 

vorpal.blade:Name=*,Type=Configuration,*

The list of Name attributes from the matching MBeans becomes the dropdown's options. Click a name and the editor reads SCHEMA.json + the current <app>.json for the selected target and renders the form.

Why an app might not appear

SymptomLikely cause
App is deployed but missing from the dropdown The app didn't call new SettingsManager(...) on startup, or the call threw an exception. Check the application's server log for stack traces during deployment.
App appears but the form is empty The Settings POJO has no @JsonPropertyDescription-annotated fields (rare), or the schema generation failed silently. Check the file-system: does config/custom/vorpal/samples/<app>/SCHEMA.json exist?
App was just deployed but doesn't appear yet MBean registration happens during contextInitialized; depending on deploy order it can take a few seconds. Refresh the dropdown.
Two clusters show the same app, but only one is editable The Configurator runs on the AdminServer and sees only MBeans that have been federated into the domain runtime. A managed server's MBean is visible; an AdminServer-only deploy of the app makes editing trivial. Cluster-only deploys also work because the cluster MBean federates up.

Inspecting the underlying MBeans

If you want to verify the MBean directly, point a JMX client at the AdminServer's DomainRuntime MBean server and query vorpal.blade:Name=*,Type=Configuration,*. Each result exposes the SettingsMXBean interface (openForWrite, write, close, reload, plus getters for the current JSON text and lastModified timestamps per scope).

Convention

The MBean Name is the app's web.xml <display-name> with anything after # stripped (the basename). Keep display names short, single-word, lowercase — that's the string operators see in the Schema dropdown.

The other path: file-based discovery

The editor also scans the on-disk config/custom/vorpal/ directory tree for JSON files. Files whose basename doesn't match any registered MBean still appear in the Files tab (for editing template files, REST body templates, LDAP queries, SQL, etc.) but they do not get form view — only the raw editor.
