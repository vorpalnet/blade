Target Scopes

Every BLADE app's configuration can be set at three scopes — domain, cluster, and per-server. Knowing which to use (and how they combine) is the single most important thing to understand before editing.

Target Scopes

The three scopes

Domain
One JSON file per app, read by every server in the domain. The base layer — almost everything you edit will be at this scope.
Cluster
One JSON file per app, per named cluster. Overrides the domain file for servers in that cluster. Use this when one cluster needs different behavior (e.g. a canary cluster with stricter logging).
Server
One JSON file per app, per server. Overrides both cluster and domain. Reserved for the rare per-server tweak — usually a debug session or a one-off observation node.

Precedence

The receiving application's SettingsManager merges all three layers at read time. Conflict resolution: Server > Cluster > Domain. Each layer is a JSON document; the merge is field-by-field, not file-replacing — a Cluster file that sets only one field overrides only that field, leaving the rest of the Domain config in place.

Mental model

Think of Domain as the default settings sheet, Cluster as a sparse override sheet, and Server as a one-off override sheet. The application sees the layered result, not the individual sheets.

On-disk layout

$DOMAIN_HOME/
└── config/custom/vorpal/
    ├── <app>.json                       ← Domain scope
    ├── _clusters/
    │   └── <cluster-name>/
    │       └── <app>.json               ← Cluster scope
    └── _servers/
        └── <server-name>/
            └── <app>.json               ← Server scope

The Target dropdown in the editor enumerates every directory it finds under config/custom/vorpal/ — the bare directory (Domain), each _clusters/<name>, and each _servers/<name>. A target only appears if its directory exists, so to add a new cluster override you must first create the cluster directory.

Choosing a scope

ScenarioRecommended scope
Production-wide setting change (new ACL entry, new route) Domain
One cluster is acting as a canary for a new behavior Cluster — set the change there and let it bake before rolling to Domain
One server is exhibiting a specific issue and needs a temporary debug log level Server — flip the logging knob, observe, then delete the file
Permanent split (cluster A handles internal calls, cluster B handles external — different rule sets) Two distinct apps, both at Domain scope. Cluster overrides become a maintenance nightmare for permanent differences.

Cleaning up overrides

To remove a Cluster or Server override, delete its JSON file from disk and the layered read drops back to the lower scope. The editor does not currently expose delete from the form; use the Configurator's Files tab to find the file, or rm it directly from the domain home.

Be deliberate with Server scope

Per-server overrides leave one server behaving differently from its peers. If you forget the override exists, traffic-anomaly debugging gets very confusing. Treat Server scope as a transient debugging tool, not a long-term configuration mechanism.
