Overview

What it does

Every BLADE service has a configuration POJO with @JsonPropertyDescription annotations on its fields. The framework generates a JSON Schema from that POJO at deploy time and writes it out alongside the current configuration JSON. The Configurator reads those schemas, builds a form for each, and gives an operator a single place to edit every service's settings without hand-editing files.

End-to-end edit flow

The journey of a single field change:

  1. Operator picks a target (Domain, Cluster, or Server scope) and a schema (one BLADE app's settings).
  2. Editor fetches the current JSON from config/custom/vorpal/[scope]/<app>.json and renders a form from the matching SCHEMA.json.
  3. Operator edits the form. The dirty indicator appears; an auto-save draft is kept in browser localStorage.
  4. Operator presses Save. The browser POSTs the updated JSON to a servlet in this WAR, which writes it to disk and timestamps the previous copy under versions/.
  5. This WAR's ConfigurationMonitor sees the file-system change event and matches it to a deployed app's SettingsMXBean by Name + scope. It opens the MBean for write, streams the new JSON line by line, closes, and calls reload().
  6. The receiving application's SettingsManager re-parses the JSON into its POJO and swaps the new current reference. Subsequent SIP messages see the new config.
Save vs. Publish

Save writes the JSON to disk and triggers the MBean update for the scope you are currently editing. Publish additionally promotes a draft into the active revision lineage — used when you want to record a deliberate milestone rather than an incremental edit.

Why JSON Schema

Every editable surface in the Configurator is generated. The form fields, the validation rules, the inspector descriptions, and the JSON-vs-Form view all derive from the same JSON Schema. Adding a new BLADE service or a new field to an existing one requires zero changes to the Configurator — only adding the annotation to the service's settings class.

This is the config-first design principle: the configuration shape is the source of truth, and the Java code falls out from it.

Where it fits in the admin tier

ConfiguratorEdits service configuration JSON. Where you spend most of your operator time.
Flow EditorEdits FSMAR routing flows (visual, not JSON-form).
LogsReads logs from every cluster node.
TuningEdits server-tier WLS / OCCAS tuning knobs (work-managers, SSO, node-manager, SIP timers).
CRUD EditorEdits service-side resource tables (CRUD service rows, not config).
