Form Layout

The Form Editor generates a labeled field for every property in a settings class's JSON Schema. You influence that layout with annotations from org.vorpal.blade.framework.v2.config placed on the settings class and its fields. At schema-generation time SettingsManager emits these as x-* hints, and the editor reads them when it renders the form. The form code is fully generic — it never hard-codes anything per schema.

The default

A field that is not placed in any group renders on its own full-width row, and fields stack top-to-bottom. You only reach for an annotation when you want to deviate from that — group fields onto one row, stack them into columns, or box them under a title.

Field-level: @FormLayout

Placed on a field or — by convention, alongside @JsonPropertyDescription — its getter. It changes how one field is rendered or edited.

FlagEffect
multiline = trueRender a <textarea> (implies wide).
password = trueMasked <input type="password">.
readOnly = trueDisabled input — visible but not editable. Good for computed/derived values. On a complex property (object/map/array) the entire subtree renders disabled.
regexTest = trueAdds a "test" button beside the field that opens the regex-tester modal.
collapsed = trueRender the property's section minimized even when it has data. For machine-managed payloads (e.g. the Flow editor's diagram layout) that operators should see but not be drawn to.
wide = trueForce the field onto a full row.
The wide gotcha

wide = true only matters for a field that is inside a @FormLayoutGroup row, where it forces that one field to break out to its own full row. An ungrouped field is already full-width, so wide does nothing there — don't add it expecting an effect.

Class-level: @FormLayoutGroup

Repeatable, placed on the class. Each annotation defines one horizontal row, in one of two mutually-exclusive forms.

Flat row

Set value with field names; they flow left-to-right as compact tiles.

@FormLayoutGroup({ "loggingLevel", "sequenceDiagramLoggingLevel",
                   "configurationLoggingLevel", "analyticsLoggingLevel" })
[ loggingLevel ] [ sequenceDiagramLoggingLevel ] [ configurationLoggingLevel ] [ analyticsLoggingLevel ]

Row of columns

Set columns instead of value. Each @FormLayoutColumn is a vertical stack of fields; the columns sit side-by-side, so the row becomes a grid of independently-stacked tiles. 

@FormLayoutGroup(columns = {
    @FormLayoutColumn({ "fileSize", "fileCount" }),
    @FormLayoutColumn({ "useParentLogging", "appendFile", "colorsEnabled" })
})
[ fileSize  ] [ useParentLogging ]
[ fileCount ] [ appendFile       ]
              [ colorsEnabled    ]

value and columns are mutually exclusive — when both are set, columns wins. The two forms emit different schema keys (x-form-groups vs x-form-columns), so introducing a column group never changes how your existing flat groups render.

Class-level: @FormSection

A titled, bordered sub-container. Fields you list render inside its box; everything else renders at the top level. Groups still apply inside a section — a section field that is also in a @FormLayoutGroup renders that group's row (flat or columns) within the section's body. 

@FormSection(title = "Connection",     fields = { "url", "method" })
@FormSection(title = "Authentication", fields = { "authentication" })

Ordering

Within a group or column, fields render in the order you list them in the annotation. Group placement and ungrouped-field order, however, follow the schema's property order, which is alphabetical unless the class carries @JsonPropertyOrder. Add that to put fields — and therefore the rows they anchor — in a deliberate order. 

@JsonPropertyOrder({
    "directory", "fileName",
    "loggingLevel", "sequenceDiagramLoggingLevel",
    "configurationLoggingLevel", "analyticsLoggingLevel",
    "fileSize", "fileCount", "useParentLogging", "appendFile", "colorsEnabled"
})

Worked example

The framework's LogParameters class combines all of the above — two flat rows, one row of columns, and an explicit property order: 

@JsonPropertyOrder({ "directory", "fileName",
    "loggingLevel", "sequenceDiagramLoggingLevel",
    "configurationLoggingLevel", "analyticsLoggingLevel",
    "fileSize", "fileCount", "useParentLogging", "appendFile", "colorsEnabled" })
@FormLayoutGroup({ "directory", "fileName" })
@FormLayoutGroup({ "loggingLevel", "sequenceDiagramLoggingLevel",
                   "configurationLoggingLevel", "analyticsLoggingLevel" })
@FormLayoutGroup(columns = {
    @FormLayoutColumn({ "fileSize", "fileCount" }),
    @FormLayoutColumn({ "useParentLogging", "appendFile", "colorsEnabled" })
})
public class LogParameters implements Serializable { ... }

Schema keys

For the curious — what each annotation emits into the generated .jschema:

AnnotationSchema keyShape
@FormLayoutGroup(value=…)x-form-groups[ ["a","b"], … ]
@FormLayoutGroup(columns=…)x-form-columns[ { "columns": [ ["a","b"], … ] } ]
@FormSectionx-form-sections[ { "title": "…", "fields": ["a"] } ]
@FormLayoutx-wide / format / x-readonly / x-regex-test / x-collapsedon the field's own schema node
