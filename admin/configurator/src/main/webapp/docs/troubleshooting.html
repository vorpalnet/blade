Troubleshooting

Symptom-to-cause table

SymptomLikely causeWhere to look
Schema dropdown is empty No BLADE apps deployed, or none registered a SettingsManager successfully Check the AdminServer log for stack traces during deploy. Also see Schema Discovery.
Schema dropdown is missing a specific app That app's startup did not call new SettingsManager(...), or the call threw App's server log around contextInitialized. Look for Unable to load <app>.json configuration file.
WebSocket status stuck on "Connecting…" Firewall/proxy strips Upgrade headers, or the WAR's WebSocket endpoint isn't reachable Browser devtools network tab — filter on websocket. The path is the WAR-relative ws upgrade endpoint.
blade-validate.sh or an API curl redirects to the login page (302) Credentials weren't sent in the Authorization header, so FORM auth took over Pass -u user:pass (or --password to the script). The API uses HTTP Basic, not the login form — see Command-Line Validation.
API curl returns 401 / 403 401: bad credentials. 403: the user authenticated but holds none of the admin roles Check the user/password (single-quote passwords with !/$). The user needs an Admin/Operator/Deployer/Monitor role in the WebLogic realm.
API curl returns 404 Wrong path — the context root is blade/configurator, not configurator Use http://host:7001/blade/configurator/api/v1/validate. The script's --context defaults to blade/configurator.
Form loads but fields are read-only / empty Schema generation failed silently; no fields means no annotations were picked up Check config/custom/vorpal/samples/<app>/SCHEMA.json on disk. If missing or trivially small, look in the app's deploy log.
Save returns success, but the running app's behavior doesn't change The file is being written, but ConfigurationMonitor isn't picking it up — possibly running on the wrong node, or the MBean federation didn't activate Verify the change appears on disk under config/custom/vorpal/[scope]/<app>.json. If yes, check the AdminServer log for ConfigurationMonitor log lines around that save time.
Validation Summary lists errors but the form looks fine A nested array element fails a constraint that isn't visible until you expand it Click an error in the summary; the editor scrolls + focuses the offending field.
Auto-save draft banner shows for a config you don't recognize A previous session was abandoned mid-edit on this target + schema If you don't want the draft: press Discard. If you do: Restore.
Two operators editing the same target overwrite each other The Configurator is last-write-wins; there is no per-cell locking Use the Diff modal before saving. Use Version History to recover an overwritten revision.

Common log signatures

Failed schema generation

java.lang.RuntimeException: Cannot generate schema for class ...
    at org.vorpal.blade.framework.v2.config.SettingsManager.build(SettingsManager.java:...)

Usually a cycle in the Settings POJO graph, or a Jackson serialization annotation that the schema generator can't reason about.

Permission failure on save

java.nio.file.AccessDeniedException: ./config/custom/vorpal/<app>.json

Filesystem-level permission problem. The user the AdminServer runs as needs write access to $DOMAIN_HOME/config/custom/vorpal/ and every _clusters/<name> and _servers/<name> subdirectory.

MBean reload exception

ConfigurationMonitor.updateManagedMBeans path=...
    javax.management.RuntimeMBeanException: SettingsManager.reload threw IOException

The JSON on disk is syntactically valid but failed to deserialize into the Settings POJO — usually a field-type mismatch after a schema change. Roll back via Version History.

If all else fails

Roll back to the last known-good revision via Version History → Restore. Always faster than debugging in production.
