Overview

What FSMAR is

FSMAR — the Finite-State-Machine Application Router — is the OCCAS-side component that decides which BLADE application gets each initial SIP request. Before any servlet sees the message, FSMAR pattern- matches the request against a state-machine descriptor and emits an ordered list of applications to invoke. The decision happens once per dialog, at the initial-INVITE (or REGISTER, SUBSCRIBE, MESSAGE, PUBLISH, REFER, OPTIONS, NOTIFY) — subsequent in-dialog messages follow the same chain by stickiness.

Why a state machine

The natural shape of "route this call" is conditional: "if From-URI matches ACL X, run the analytics service then the proxy-router; if it matches Y, run the hold service first." Expressing that as a single deeply-nested rule list is brittle. Expressing it as a directed graph of states and transitions reads cleanly and composes well — adding a new branch is adding a new edge, not refactoring the rules tree.

What the editor produces

An FSMAR XML descriptor. That descriptor ships inside the FSMAR JAR (blade-fsmar.jar or blade-fsmar.jar) at $DOMAIN_HOME/approuter/. Activation in OCCAS is via the admin console; the engine tier reboots to pick up the new descriptor.

The editor reads the descriptor on load, lets you edit it visually, and writes it back on Export. Round-trip is via two servlets in this WAR: FsmarImportServlet (XML → mxGraph cells) and FsmarExportServlet (cells → XML).

Two FSMAR generations

The framework ships two FSMAR libraries side by side: FSMAR 2 (legacy, kept for backward compatibility) and FSMAR 3 (current, ongoing development). They share zero code and are activated one-at-a-time via the OCCAS admin console. The editor works with either; pick the descriptor format your domain currently runs.

Where it fits in OCCAS request handling

  1. SIP message arrives at the engine tier.
  2. OCCAS asks the active SipApplicationRouter (FSMAR) to compute the application chain.
  3. FSMAR runs its state machine against the message and returns the chain.
  4. OCCAS invokes each application in order; the dialog is bound to that chain for its lifetime.

What's NOT in the editor

  • Per-application configuration (lives in the Configurator).
  • Cluster topology / server tuning (lives in the Tuning app).
  • Active call inspection / live state (FSMAR is stateless per-request; there is no "current calls" view here).
