FSMAR Concepts

The four primitives

State
A node in the graph. Represents either an application invocation (e.g. "run analytics") or a control point (start, end, decision). Rendered as a labeled shape.
Transition
A directed edge between two states, evaluated under its trigger's SIP method. Carries an optional when condition; transitions are tried in order and the first match wins.
Trigger
An event that activates the start of a flow — typically a SIP method on an initial request. Triggers live on the edge from the Ingress cloud.
Condition
A predicate evaluated against the in-flight message. Boolean expressions over message attributes (From-URI, To-URI, headers, etc.). A transition with a condition only fires when the condition is true.

Ingress and egress

Every flow has two terminal clouds:

  • Ingress — the entry point. Transitions out of Ingress carry triggers that match incoming SIP messages.
  • Egress — the exit point. When the flow walks to Egress, FSMAR returns the accumulated application chain to OCCAS.

Both clouds round-trip to FSMAR XML as the literal string null — there's no Ingress or Egress node per se; the clouds are visual stand-ins for "start of the chain" and "end of the chain."

Initial vs. non-initial requests

FSMAR only consults the descriptor on the initial request of a SIP dialog. The methods that can be initial — and therefore the methods that can appear in a trigger — are:

MethodTypical use
INVITEEstablish a call
REGISTEREndpoint registration
SUBSCRIBEEvent subscription (BLF, dialog-info, etc.)
OPTIONSCapabilities query / keepalive ping
MESSAGESIP IM
PUBLISHPresence publication
REFEROut-of-dialog transfer
NOTIFYOut-of-dialog notify (rare)

In-dialog messages (ACK, BYE, re-INVITE, in-dialog NOTIFY, etc.) follow the chain established for the initial — FSMAR is not re-consulted.

Application chains

FSMAR's return value is an ordered list of application names. OCCAS invokes them in order; each application decides what to do with the message and either passes it along or terminates the chain. A typical chain for an inbound INVITE might be: 

[ acl → analytics → proxy-router ]

Each application's name in the chain corresponds to a deployed BLADE service. The chain is constructed by walking the flow from Ingress to Egress, accumulating state names along the way.

Conditions

Conditions are expressions over the inbound message. The canonical syntax depends on FSMAR version (2 versus 3). Common building blocks:

  • Method match: method == "INVITE"
  • URI regex: from.uri ~= ".*@trusted-domain\\.com"
  • Header presence: headers["X-Customer-ID"] != null
  • Boolean composition: cond1 && (cond2 || cond3)
Naming convention

Name states after the application they invoke (e.g. acl, proxy-router). Name transitions after the condition they represent (e.g. "trusted realm", "emergency-call"). Future-you will thank present-you for the descriptive labels.
