FSMAR 3, feature by feature

FSMAR is the brain that sits in front of every call and decides which of your SIP applications handle it, and in what order. FSMAR 3 makes that brain a data-driven state machine: it extracts facts from the call, remembers them across every hop, decides with readable expressions, and builds the routes from what it learned — and it lets you watch the whole thing happen before a single real call is placed.

This tutorial walks that capability set one chapter at a time. Every chapter ships a small, self-contained config. Load it, read the one or two states it adds, then run the suggested calls in the Route Simulator and watch the path light up.

Before you begin

Open the Editor. For each chapter, download its config (the links below), then in the editor use Open / Load → Choose file… to import it. After loading, click Auto-position then Fit for a clean layout, and open the Route Simulator (the play-head toolbar button) to run calls. All example callers and domains are fictional (example_co.com).

Chapter 1 — The data-driven core

Why it mattered. In the old per-subscriber model, "treat Bob differently" meant a dedicated rule with Bob's proxy URI written in by hand. A hundred VIPs meant a hundred rules, and the route was a literal string baked into each one.

What FSMAR 3 does. Two regex selectors capture the caller and callee user-parts into the routing context (${From.user}, ${To.user}). One transition routes the VIP bob to a priority proxy; a second handles everyone else with a single templated route, sip:${To.user}@proxy.example_co.com. The callee's user-part, captured at the front door, is still available one hop later at b2bua to build the registrar route — that is capture-and-carry: FSMAR keeps the facts it extracted for the whole call.

Try it — 01-basic-routing.json

Load it, then click the null entry state → Selectors panel to see the two regex selectors. Click the INV-vip edge to read when: ${To.user} == 'bob' and its route. In the Route Simulator: Method INVITE, To sip:bob@example_co.com → Run — watch the selectors fill and INV-vip fire. Change To to sip:carol@example_co.com → Run → INV-everyone fires with the same templated route, a different callee.

Chapter 2 — Smart conditions

Why it mattered. Time-of-day routing, percentage rollouts, emergency detection — none of that fit the old equality/regex header tests.

What FSMAR 3 does. A full, safe operator set — == != < > <= >=, && || !, matches (full-string regex), contains (substring), and insubnet (CIDR) — plus pseudo-variables the engine publishes on every hop. This config fans out from null: 911 → emergency; toll-free regex → IVR; weekend or after-hours → voicemail; ${hash100} < 5 → a canary build (a stable 5% of calls); business hours → live agents.

Try it — 02-conditions.json

Click a few edges to read conditions like ${hour} < 9 || ${hour} >= 17 and ${hash100} < 5. In the Route Simulator, use the pseudo-variable overrides: set hour = 3, dayOfWeek = TUESDAY → Run → after-hours → voicemail. Set hour = 12, hash100 = 2 → the canary path. Set hash100 = 73 → live agents. Set To sip:18005551234@example_co.com → toll-free wins (it is matched higher up).

Chapter 3 — Tiering as data

Why it mattered. Gold / silver / bronze used to mean editing routing rules every time a customer changed plan.

What FSMAR 3 does. A table selector classifies the caller through a translation table — the tiers are table rows an operator edits, not code. The routing just tests ${tier}. In the editor this renders as a plan dispatch family: the edges are badged INVITE · tier=gold, tier=silver, tier=bronze, and an unconditional default. Tables also support longest-prefix matching (dial plans, via a trie) and range matching ("8-17"), not just exact.

Try it — 03-tiering.json

Click the null state → Selectors → open the customerTier table to see the rows (alice → gold, carol → gold, bob → silver, dave → bronze). In the Tasks window, use View filter → Plans and uncheck all but gold to isolate that one path. In the Route Simulator: From sip:alice@… → gold team; From sip:dave@… → bronze; From sip:zoe@… (no row) → the general queue via the default.

Chapter 4 — Ingress, egress, and capture-and-carry

Why it mattered. The old model had no first-class notion of where a call entered or how it leaves, and once an app rewrote a header the original value was gone.

What FSMAR 3 does. Three things at once:

  • Ingress — entry states classified by source subnet (${originIP} insubnet '10.20.0.0/16'), drawn as clouds with a down-arrow.
  • Egress — two kinds, distinguished by topology, not a stored flag. ROUTE_FINAL has no line out: the call leaves OCCAS for the carrier and never returns. ROUTE_BACK has a dashed line back to a state: the call detours (say, to a media server for a greeting) and then resumes at that state. Both draw as clouds with an up-arrow.
  • Capture-and-carry — screening may rewrite From; we re-capture it as ${callerNow.user}, but the original ${From.user} from the front door is still there, carried across every hop.
Try it — 04-ingress-egress.json

Name the shapes: ingress clouds (Atlanta / Dallas, down-arrow), the carrier egress (solid, ROUTE_FINAL), and the media-greeting egress (dashed line back to b2bua, ROUTE_BACK). In the Route Simulator, set the origin IP to 10.20.5.9 → the call enters through the Atlanta cloud. From sip:anonymous@… → it detours out the media-greeting egress and returns to b2bua. To sip:12125551234@… (an E.164 number) → it exits to-carrier and is gone.

Chapter 5 — One application, invoked twice

Why it mattered. When a state was its application, you could never run the same app twice in one call, and a reused app was one tangled box.

What FSMAR 3 does. A state's id is separate from its application. Two states, b2bua-caller and b2bua-callee, both run the b2bua app — so the B2BUA runs once for the originating leg and again for the terminating leg, with different routing (and a different JSR-289 region) each time. In the editor they are two clean boxes, each labelled b2bua with its id underneath.

Try it — 05-b2bua-twice.json

Click each box: the State ID reads b2bua-caller / b2bua-callee, while the application Name is b2bua for both. In the Route Simulator, run an INVITE and watch the call visit b2bua-caller then b2bua-callee — two distinct hops, one application.

Chapter 6 — The whole switchboard

What FSMAR 3 does. FSMAR routes every initial SIP method, not just INVITE: REGISTER → registrar, SUBSCRIBE / PUBLISH → presence, OPTIONS → responder, MESSAGE → IM relay, INVITE → its own logic. The default application is the safety net: here INVITE has only a toll-free rule, so any other INVITE matches nothing and falls to defaultApplication (b2bua). The simulator can also mark an app undeployed to show bypass — the router skips an undeployed app and keeps evaluating, so you can roll out a chain before every piece is live.

Try it — 06-multimethod.json

In the Route Simulator: Method REGISTER → registrar; SUBSCRIBE → presence; MESSAGE → IM relay. Method INVITE, To sip:bob@… (not toll-free) → "no transition matched → default application b2bua"; point at the decision line. Re-run an INVITE with a downstream app marked undeployed to watch the bypass.

Chapter 7 — Reaching into the message

Why it mattered. The old model read headers. That was all.

What FSMAR 3 does. Selectors can reach the body: sdp (route video calls to a video bridge by reading the m= line), json (JsonPath into a JSON body), and xml (XPath). Plus magic-URI routing: a Request-URI of sip:hold (no user-part, host = an application name) routes straight to the hold application, bypassing the state machine — handy for feature codes and park / retrieve.

Try it — 07-snippets.json

Click null → Selectors to see the sdp, json, and xml selectors side by side. In the Route Simulator, toggle paste a real INVITE and paste one whose SDP has an m=video line → Run → video-call fires to the video bridge. Swap to an audio-only SDP → audio bridge. Set the Request-URI to sip:hold → it routes directly to the hold app (which must be deployed).

Chapter 8 — See it run

What FSMAR 3 does. The previous chapters, combined into one production-shaped flow — and the thing the old model never had: you can watch it. On the full config, the observability suite is:

  • Route Simulator — synthetic calls, paste a real INVITE, pseudo-variable overrides, and mark-undeployed to explore bypass / fallback, all before deploying.
  • Capture & replay — arm captureNextCalls(n) on the metrics MBean; the engine records real routing traces in the simulator's own format; replay them on the diagram.
  • Live heat overlay — the editor polls per-transition hit counts across every engine and paints them on the edges: count labels, with stroke width scaled by traffic share.
  • Metrics MBean (org.vorpal.blade:type=Fsmar3,name=metrics) — requests routed, default-app fallbacks, undeployed bypasses, cycle detections.
  • Load-time validation — a malformed when is flagged SEVERE at load (it would otherwise silently never match); the editor surfaces the same findings above the exported JSON.
Try it — 08-everything.json

Load → Auto-position → Fit to see the full enterprise flow (ingress clouds, tiering, the B2BUA legs, both egress kinds). In the Route Simulator, run a few representative calls (a gold caller, an anonymous caller through the media greeting, an off-net E.164 to the carrier). Flip to Replay / Live to show real traffic on the same diagram. Then Export and look at the validation findings; break one when (e.g. ${tier} ==), re-export, and watch the SEVERE finding appear.

Appendix B — verified syntax reference

Operators
== != < > <= >= && || ! matches contains insubnet. matches is a full-string Java regex; contains is a literal substring; insubnet is real CIDR math (IPv4 / IPv6). There is no startsWith / endsWith.
Pseudo-variables
Published every hop: ${method} ${requestURI} ${directive} ${region} ${previousApp} ${previousState} ${hour} ${dayOfWeek} ${hash100}. ${hash100} is a stable 0–99 bucket derived from the Call-ID.
Selector types
attribute regex json xml sdp table.
Table strategies
hash (exact, default), prefix (longest-prefix, trie — dial-plan semantics), range ("lo-hi" integer interval).
Egress kind
Topology, not a stored flag: a line back to a state = ROUTE_BACK (resume there); no out-edge = ROUTE_FINAL (leaves for good).
Magic URI
sip:<appname> (null user-part, host = application name) routes straight to that app with NO_ROUTE.
