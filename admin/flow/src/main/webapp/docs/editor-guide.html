Editor Guide

Layout

The editor uses mxGraph (jgraphx). The viewport is divided into:

Canvas
Main drawing surface — grid background, panning with middle-mouse or right-drag, zoom with wheel + Ctrl.
Palette
Left rail with stencils (states, decision shapes, Ingress/Egress clouds). Drag onto the canvas to add.
Tasks window
Floating panel with filtered overviews — lists every state, every transition, and every condition in one scrollable list with click-to-jump.
Inspector
Side panel that updates with the selected cell — edit name, condition, trigger, and FSMAR-specific attributes there.

Common actions

Adding a state

  1. Drag a stencil from the palette onto the canvas. The default stencil is an application-invocation state.
  2. Double-click to edit the label. The label becomes the application name that ends up in the application chain.

Adding a transition

  1. Hover over a source state's edge — handles appear.
  2. Drag from a handle to the target state. A directed edge is drawn.
  3. Double-click the edge to set its label (typically the human-readable condition name).
  4. Open the Inspector to set the actual condition expression and (for Ingress-originating edges) the trigger method.

Editing a trigger

Triggers attach to the transition out of Ingress. Select that transition; in the Inspector, set the SIP method (INVITE, REGISTER, etc.) and any additional match criteria. The method dropdown is constrained to the methods FSMAR supports — see FSMAR Concepts.

Editing a condition

Conditions attach to any transition. Select the transition; in the Inspector, type or paste the condition expression. Syntax depends on FSMAR version. The Tasks window's condition list is the fastest way to audit every condition in the flow without clicking each edge.

Tasks window

The Tasks window is a flat overview of the otherwise deep graph. Useful for:

  • Finding orphan states (no incoming transitions).
  • Auditing every condition for a typo or stale predicate.
  • Jumping to a transition by name when you can't find it on the canvas.

Filter panel inside Tasks lets you scope by cell type (state / transition / condition) and by name regex.

Service plans (tier dispatch)

The gold/silver/bronze pattern has two halves, and they deliberately live in different places:

  • Classification is data. A table selector on the Ingress (or any state) maps a caller attribute to a plan value — e.g. key ${From.user}, rows alice → tier=gold. Operators edit rows in the Configurator; no diagram change, no redeploy.
  • Dispatch is edges. One transition per plan with a mutually exclusive condition (${tier} == 'gold', 'silver', …) plus one unconditional default. Because no two conditions can both match, first-match-wins evaluation order cannot change the outcome.

The editor makes this pattern first-class, with no change to the FSMAR JSON:

  • Badged edges — a dispatch condition renders as INVITE · tier=gold; the unconditional member of a family as INVITE · default. Display only; hand-written configs that follow the pattern render identically.
  • Add plan dispatch… — a button on the State/Ingress properties panel generates the whole family (conditions, evaluation order, target states created if missing) in one undo step.
  • Plans view filter — when dispatch conditions exist, the View panel grows a Plans section; uncheck everything but tier=gold to show a customer exactly what the gold path does.
  • Shadowing validation — export-time findings flag unreachable transitions (an unconditional one anywhere but last) and duplicate conditions, the two ordering mistakes a canvas cannot show.

Property panel templates

The properties panel (in the Tasks window) is driven by three HTML templates, loaded once and switched by what's selected on the canvas:

state.html
State / Ingress / Egress panel — name, selector editing, and the "Add plan dispatch…" button.
transition.html
Transition panel — SIP method, when condition, evaluation order, subscriber, region, routes, route modifier.
config.html
FlowModel panel (shown when nothing is selected) — defaultApplication.
Save vs. Export

Save downloads the diagram as mxGraph XML — your drawing, including layout; reload it later with Open. Export produces the FSMAR 3 JSON (validated, with findings shown) — the deployable artifact. Neither writes anything server-side; applying the JSON to a cluster goes through the Configurator.
