Import / Export Round-Trip

Two servlets, one descriptor

FsmarImportServlet
GET endpoint. Reads the current FSMAR XML from the deployed JAR and translates it into mxGraph cells the editor consumes.
FsmarExportServlet
POST endpoint. Accepts the editor's serialized graph, translates it back to FSMAR XML, and writes the result.

What happens on Import

  1. Browser hits the Import endpoint.
  2. Servlet locates the active FSMAR JAR under $DOMAIN_HOME/approuter/ and extracts the descriptor XML.
  3. Each state element in the XML becomes an mxCell with the state's name as the cell's user-object value.
  4. Each transition element becomes a directed edge with source and target resolved by name.
  5. The Ingress and Egress clouds are synthesized — there is no XML element for them; they are visual stand-ins for the start and end of every chain.
  6. The Tasks window is populated from the same data.

What happens on Export

  1. Editor serializes the current graph to mxGraph XML and POSTs it.
  2. Servlet walks the graph, identifies states and transitions, and emits the FSMAR XML element tree.
  3. The synthesized Ingress and Egress clouds are dropped (they round-trip as the literal string null in the descriptor).
  4. The result is written back to the FSMAR JAR (or to a staging file, depending on configuration).
Reboot to activate

FSMAR is loaded by OCCAS via SPI at engine-tier startup. A new descriptor does not take effect until the engine-tier servers are restarted — see the project DEPLOYMENT.md for the rolling-reboot sequence.

XML shape (FSMAR 2)

An FSMAR 2 descriptor is essentially a list of state-transition tuples wrapped in a header. Simplified example: 

<fsmar>
  <state name="start"/>
  <state name="acl"/>
  <state name="proxy-router"/>

  <transition from="null" to="acl">
    <trigger method="INVITE"/>
  </transition>

  <transition from="acl" to="proxy-router"/>
  <transition from="proxy-router" to="null"/>
</fsmar>

Note that from="null" is the Ingress cloud and to="null" is Egress — the same literal string in both places.

XML shape (FSMAR 3)

FSMAR 3 adds richer condition semantics and clean-up of the legacy idioms. It is not source-compatible with FSMAR 2; the editor detects which generation a descriptor uses from the root element and adjusts its export accordingly.

Edits that don't round-trip cleanly

The visual editor's coordinate / size information for each cell is preserved on round-trip in a separate side-car region of the descriptor. Pure-text edits to the descriptor (hand-editing the XML outside the editor) will lose the layout — re-importing in the editor produces an auto-layout. Avoid manual XML edits unless you're prepared to re-arrange the canvas after re-importing.
