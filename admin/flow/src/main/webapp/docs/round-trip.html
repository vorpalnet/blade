Import / Export Round-Trip

Three servlets, one JSON configuration

FsmarImportServlet /fsmarImport
POST. Accepts FSMAR 3 JSON (pasted into the Import dialog) and returns the mxGraph XML model the editor renders.
FsmarExportServlet /fsmarExport
POST. Accepts the editor's serialized graph and returns FSMAR 3 JSON for download or copy. Nothing is written server-side.
FsmarValidateServlet /fsmarValidate
POST. Semantic checks on the JSON — run automatically on every export; findings appear above the JSON, labeled ERROR (won't route as written: unparseable when, bad region/routeModifier, regex that doesn't compile, transition with no next), WARNING (probably a mistake: unknown field names, unreachable states), or INFO. The when check uses the same expression parser the router itself compiles with.

What happens on Import

  1. Paste FSMAR 3 JSON into the Import dialog.
  2. Each entry in states becomes a State vertex; its selectors ride along on the vertex (edit them in the State properties panel).
  3. The "null" state becomes the Ingress cloud — its selectors (run against initial requests) live on the cloud. Transitions targeting "null" get an Egress cloud.
  4. Each transition becomes a directed edge labeled with its SIP method, carrying when, subscriber, region, routes, and routeModifier — plus a sequence number, because transitions in a trigger are evaluated first match wins and edge order on a canvas is meaningless without it.
  5. A config the diagram cannot represent faithfully (e.g. a transition with no next) is rejected with a named reason — never imported lossily.

What happens on Export

  1. Editor serializes the current graph and POSTs it; the servlet rebuilds the FSMAR 3 JSON tree.
  2. Ingress and Egress clouds round-trip as the literal string null.
  3. Transitions are emitted in evaluation order (their sequence numbers), grouped by state and SIP method.
  4. Validation runs automatically; findings appear above the JSON.
  5. Download or copy the JSON and apply it like any FSMAR 3 config change. A dangling edge (not connected at both ends) fails the export with a named reason.

The no-silent-loss guarantee

Every field of the imported JSON survives export — including fields this editor has no UI for. Anything not explicitly modeled (TableSelector tables, XML namespaces, fields added by future FSMAR versions) is carried through verbatim in passthrough attributes and merged back on export. A full-featured config round-trips to JSON-equality; the regression gate is FsmarRoundTripSmokeTest in the flow module's test sources.

Two harmless normalizations: every state in the export carries an explicit "triggers" object (matching how FSMAR 3 itself serializes), and a state vertex that exists only on the canvas exports as an empty state entry.

Pre-rework diagram files

Diagram XML files saved before the 2026-06 rework stored conditions in transition-level "selector groups" — a shape that never matched FSMAR 3. Exporting such a diagram is refused with a named reason: re-import your FSMAR configuration JSON to rebuild the diagram, then re-apply edits.

Reboot to activate

FSMAR is loaded by OCCAS via SPI at engine-tier startup. A new descriptor does not take effect until the engine-tier servers are restarted — see the project DEPLOYMENT.md for the rolling-reboot sequence.

FSMAR 2

The editor targets FSMAR 3 JSON only. FSMAR 2 (the legacy router, retained for backward compatibility) uses a different configuration shape that this editor does not read or write.

What does NOT round-trip: layout

Canvas positions are not stored in the FSMAR JSON — importing always produces an auto-layout. If you care about a hand-arranged canvas, save the diagram itself (the toolbar's mxGraph XML save) alongside the exported JSON; the JSON is the deployable artifact, the XML is your drawing.
