Routing and health, per node

The proxy-balancer service distributes initial INVITEs across tiered pools of named endpoints. Endpoints are defined once in the configuration's registry; plans are ordered arrays of tiers, selected by the request URI's host (exact match, then the longest *.suffix wildcard, then the "*" default). A tier that fails hands the call to the next tier — order tiers cheapest-first for least-cost routing.

Tier strategies

  • parallel — race every endpoint at once; the first 2xx wins and the losing legs are canceled.
  • serial — hunt one at a time in listed (priority) order.
  • random — hunt in random order: equal-cost distribution.
  • roundrobin — hunt from a rotating offset. Each node rotates independently; the aggregate stays evenly balanced.
  • weighted — hunt in smooth weighted round-robin order: over any totalWeight consecutive calls on a node, each endpoint gets exactly weight first attempts.

Failover is response-classified: route-level failures (408/timeout, 480, 404, 5xx, 3xx) escalate to the next tier; user-state and auth responses (486 busy, 401/407, 487 canceled, all 6xx) are relayed to the caller and never touch a more expensive tier.

How endpoint health is decided

Each engine node keeps its own health map — there is no shared cluster state. Two inputs write it; routing reads it.

  • OPTIONS pings — every health.pingInterval seconds (default 60) the node sends OPTIONS to every registry endpoint (unless it opts out with ping: false). ANY final response except 408/503 proves the box alive — a 405 is an endpoint that dislikes OPTIONS, not a dead one. The 408 generated when nothing answers, or a 503, marks it DOWN. Publishing a config reschedules the cycle immediately.
  • Live call legs — a 2xx from an INVITE leg marks its endpoint UP; a 503 (the SIP overload signal) marks it DOWN, honoring Retry-After as a timed backoff, else the configured health.defaultBackoff. User responses (486 busy, 603 decline, other 4xx) say nothing about the endpoint and are ignored.

Routing skips DOWN and drained endpoints when building a tier's forks; a tier with nothing routable is skipped; if no tier has a routable endpoint the caller gets 503. A BACKOFF endpoint becomes eligible the moment its window expires — the next call probes it; a repeat 503 renews the backoff. Health survives a config publish: the endpoint's registry name is its stable identity, so a name that persists keeps its status, observation history, and traffic counters; only new names start fresh (up).

Beyond the instantaneous verdict, every node keeps a bounded ring of recent observations per endpoint — each OPTIONS ping records its round-trip time, and passive call marks are logged alongside — plus live traffic counters (attempts / successes / failovers per INVITE leg). These feed the sparklines, heartbeat strips, and traffic-share bars below.

Sites and the map

The optional top-level sites registry names physical locations — datacenters, POPs, regions — with a label and decimal lat/lon; endpoints join a site via their site field, and clusterSite says where this OCCAS cluster runs. The Map view places each site geographically: when every placed site falls inside the United States it uses an Albers USA projection (with the customary Alaska and Hawaii insets); otherwise a world projection. A site without coordinates isn't lost — it collects in the Unplaced tray beside the map, as do endpoints with no site at all.

  • Site glyph — shape carries the rollup: a solid circle when every endpoint is up, a diamond when degraded (some backoff/drained/unknown), a square when something is down or undefined. The n/m UP count is always printed beside it.
  • Traffic arcs — drawn from the cluster site to each endpoint site; an arc animates (dashed, moving) when calls flowed to that site since the last poll.
  • Click a site to jump to the Topology view filtered to it (other endpoints dim).

The three views

  • Map — the geographic rollup above: one glyph per site, arcs showing live traffic.
  • Topology — the call path as a pipeline: engine nodes, then each tier in failover order. Endpoints render as cards: per-node status glyphs (one per engine — disagreement stays visible), an OPTIONS round-trip sparkline, a heartbeat strip of recent observations (short bar = up, full-height bar = down), a traffic-share bar (that endpoint's slice of the tier's attempts — watch a weighted 3:1 split prove itself), and attempt/success/failover counters. A card flashes when it took calls since the last poll. A tier with more than 12 endpoints starts as a one-card summary — counts by state plus the currently active endpoints — and expands on demand.
  • Grid — the dense table: rows are tier/endpoint, one column per engine node.

Clicking any endpoint (card or grid row) opens the detail panel: per-node status and round-trip times, the overlaid RTT trend, counters, and a time-stamped log of the most recent observations from every node.

Reading the dashboard

  • ● UP — routable; last observation succeeded.
  • ■ DOWN — skipped by routing on that node.
  • ◆ BACKOFF ns — 503'd; eligible again in n seconds (Retry-After, or the configured defaultBackoff).
  • ▢ DRAINED — administratively disabled (enabled: false): no new calls, still monitored.
  • ! UNDEFINED — a tier references an endpoint name missing from the registry; fix the configuration.
  • ? UNKNOWN — the node has no health entry for that endpoint (usually a just-published config).

Columns are engine nodes; each is that node's independent opinion. One column disagreeing with the rest points at the network path between that engine and that endpoint, not at the endpoint itself.
